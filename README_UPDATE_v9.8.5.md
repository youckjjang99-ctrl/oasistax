# OASIS CRM v9.8.5 적용 안내

## 적용 기준

- v9.8.4가 적용된 프로젝트에서 실행하는 PATCH입니다.
- ZIP 내부 파일을 기존 `정책자금자동화` 폴더에 바로 덧붙입니다.
- 별도의 상위 폴더를 만들지 않았습니다.

## 수정 목적과 영향 범위

- `public_data_api.py`: 전년 동월 API 재조회가 모든 후보를 탈락시키던
  경로를 후보 필터에서 제거하고, 기간별 API의 최근 월 순취득을 고용
  신호로 사용합니다.
- `prospect_collection_service.py`: 전년 동월 변동(축적된 경우)과 최근 월
  순취득을 하나의 고용 증가 신호로 묶어 우선순위를 정합니다.
- `prospect_db_repository.py`: 현재 가입자수를 월별 스냅샷으로 저장하고
  전년 동월 스냅샷을 일괄 불러옵니다.
- `prospect_db_center.py`: 고용 기준 선택 UI와 법정동 코드 입력란을
  제거하고 통합 고용 증가 신호 UI로 단순화합니다.
- 기존 CRM, 고객DB, 영업후보DB, 검색이력, 메모, 엑셀 기능과 `.git`은
  변경하거나 삭제하지 않습니다.

## 왜 900건 중 0건이었는가

- v9.8.4는 기본조회에 과거 `자료생성년월`을 요청해 전년 동월 사업장을
  다시 찾는 방식이었습니다.
- 실제 국민연금 응답에서는 이 과거 스냅샷을 안정적으로 반환하지 않아
  전년 자료가 없다는 판정으로 후보가 일괄 제외됐습니다.
- v9.8.5는 전년 동월 값이 없다는 이유만으로 후보를 제외하지 않습니다.
  최근 월 순취득이 증가한 사업장을 즉시 찾고, 전년 동월 값은 앞으로
  CRM이 축적한 스냅샷이 있을 때 함께 표시합니다.

## 적용 방법

1. ZIP 내용을 기존 프로젝트 폴더에 그대로 풉니다.
2. `RUN_UPDATE_v9.8.5.bat`를 실행합니다.
3. `UPDATE_OK`, `PY_COMPILE=OK`, `RUNTIME_SMOKE_TEST=OK`를 확인합니다.
4. Supabase SQL Editor에서 `supabase_v985_employee_snapshots.sql` 전체를
   실행하고, 경고창이 나오면 `Run and enable RLS`를 선택합니다.
5. Railway가 GitHub main의 새 커밋을 배포한 뒤 `DB발굴`을 확인합니다.
6. 동봉된 Git 명령어로 GitHub main에 반영합니다.

v9.8.4의 검색이력·메모 SQL은 그대로 유지하며, 이번 SQL은 전년 동월
비교를 위한 새 스냅샷 테이블만 추가합니다.

## 화면 변경

- `고용 증가 판단 기준` 선택 상자를 제거합니다.
- `시·군·구 법정동 코드`, `읍·면·동 법정동 코드` 입력란을 제거합니다.
- `고용 증가 신호 사업장만 표시` 하나로 통합합니다.
- 통합 신호는 다음 중 하나라도 성립하면 참입니다.
  - 전년 동월 가입자 증가가 CRM 스냅샷에서 확인된 경우
  - 최근 월 `신규취득자수 - 상실가입자수`가 1명 이상인 경우
- 전년 동월 스냅샷이 아직 없는 업체는 `스냅샷 축적 전`으로 표시하며,
  최근 월 순취득 증가가 있으면 정상적으로 후보에 포함됩니다.
- 매월 DB발굴을 실행할 때 현재 가입자수가 자동 저장됩니다. 같은 월의
  데이터가 1년 누적되면 전년 동월 증감도 통합 신호에 자동 반영됩니다.

## 참고

국민연금 API는 신규취득자수에 납부재개, 상실가입자수에 납부예외가
포함될 수 있어 실제 입·퇴사 인원과 다를 수 있습니다. 따라서 CRM은
이를 ‘영업 우선순위 신호’로 사용하고 최종 지원금 대상 여부는 기업의
실제 고용자료로 추가 확인합니다.

- https://www.data.go.kr/data/3046071/openapi.do?recommendDataYn=Y
- https://www.data.go.kr/bbs/ntc/selectNotice.do?originId=NOTICE_0000000004065
