# OASIS CRM v9.8.4 누적 교체 패치 적용 안내

## 적용 기준

- v9.8.0, v9.8.1, v9.8.2 또는 v9.8.3에서 적용할 수 있는 누적
  PATCH입니다.
- 이전에 전달된 v9.8.4를 적용하지 않았다면 이 파일만 사용합니다.
- v9.8.3의 고용 증가 오판정 수정과 기존 v9.8.4 기능을 모두 합친
  재작성본입니다.
- 압축 내부 파일을 기존 프로젝트 폴더에 그대로 덧붙여 사용합니다.
- 별도의 상위 패치 폴더를 만들지 않았습니다.

## 수정 파일과 영향 범위

- `app.py`: `DB발굴` 메뉴명·위치·기존 세션 호환
- `prospect_db_center.py`: 고용 기준, 검색 이력, 업종 필터, 엑셀, 메모 UI
- `prospect_collection_service.py`: 고용자료 조회·판정 및 업종 필터
- `public_data_api.py`: 기간별 조회와 전년 동월 기본·상세조회 연동
- `sales_intelligence.py`: 선택한 고용 근거를 영업주제·스크립트에 반영
- `prospect_db_repository.py`: 고용 근거, 검색 이력, 메모 Supabase 연동
- `supabase_v984_db_discovery.sql`: 추가형 테이블·컬럼 생성
- 기존 CRM 업무 화면과 기존 고객·영업 데이터에는 삭제 작업이 없습니다.

## 적용 방법

1. 압축파일 내용을 기존 `정책자금자동화` 폴더에 그대로 풉니다.
2. `RUN_UPDATE_v9.8.4.bat`를 실행합니다.
3. `UPDATE_OK`, `PY_COMPILE=OK`, `RUNTIME_SMOKE_TEST=OK`를 확인합니다.
4. Supabase 대시보드의 `SQL Editor`에서 `New query`를 누릅니다.
5. 프로젝트 루트의 `supabase_v984_db_discovery.sql` 전체 내용을
   붙여넣고 `Run`을 한 번 누릅니다.
6. Railway 배포가 완료되면 `DB발굴`의 검색 이력과 메모 저장을
   확인합니다.
7. 동봉된 Git 명령어로 GitHub main에 반영합니다.

## 기능 사용법

- `DB발굴`은 좌측 `홈` 바로 아래에서 모든 사용자가 이용합니다.
- `고용 증가 판단 기준`에서 다음 중 하나를 선택합니다.
  - `전년 동월 대비 가입자수 증가`: 현재와 12개월 전 가입자수를 비교
  - `최근 월 신규취득−상실 증가`: 기간별 API의 월간 순취득을 비교
  - `고용 증가 필터 사용 안 함`: 고용자료와 무관하게 연락처 검색
- `선택한 기준의 고용 증가 사업장만 표시`를 켜면 증가값이 1명 이상으로
  확인된 사업장만 연락처 검색 대상으로 사용합니다.
- 고용자료가 누락되면 0명으로 간주하지 않고 `확인 불가`로 표시합니다.
- 전년 동월 대비는 후보마다 과거 기본·상세조회가 추가되므로 최근 월
  기준보다 시간이 더 걸릴 수 있습니다.
- 검색을 완료하면 `내 검색 페이지 이력`에 사용자별 조회 구간이
  자동 저장됩니다.
- 최근 검색의 다음 페이지가 새 검색의 시작 페이지로 자동 제안되며
  사용자가 직접 다른 구간으로 수정할 수 있습니다.
- 업종 필터를 비워두면 전체 업종, 선택하면 해당 분류만 검색합니다.
- 검색 결과와 저장된 영업후보는 각각 엑셀로 받을 수 있습니다.
- 저장 목록의 `업체 메모`만 편집한 후 `변경한 메모 저장`을 누릅니다.
- 메모는 공용 영업DB에 저장되므로 다른 사용자도 같은 메모를 봅니다.

## v9.8.3에서 0건이 나온 이유

- 기본조회와 상세조회에는 최근 신규취득자수·상실가입자수가 포함되지
  않는데 기존 코드가 누락값을 각각 0명으로 변환했습니다.
- 그 결과 실제 고용자료를 확인하지 않았는데도 모든 사업장이
  `0 - 0 = 0명`으로 계산되어 증가 필터에서 제외됐습니다.
- 수정판은 최근 월 수치는 기간별 조회에서만 가져오고, 전년 대비는
  동일 사업장의 전년 동월 가입자수를 별도 확인합니다.

## SQL 실행 전 동작

- 기존 DB발굴, 전화번호 검색, 후보 저장 및 저장목록 조회는 유지됩니다.
- 사용자별 검색 이력 자동 저장과 업체 메모 저장만 SQL 실행 후
  활성화됩니다.
- 새 검색 기준 이력을 저장하려면 수정된 SQL의 `growth_basis` 컬럼도
  필요합니다.

## 백업과 롤백

- 적용 전 수정 대상은 `_update_backups/before_v9.8.4`에 자동
  백업됩니다.
- 적용 또는 자체검사 실패 시 파일이 자동 롤백됩니다.
- Supabase SQL은 컬럼·테이블을 추가하기만 하며 기존 데이터를
  삭제하지 않습니다.
