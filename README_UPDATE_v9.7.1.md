# OASIS CRM v9.7.1 업데이트 안내

## 수정 목적

국민연금 사업장 후보를 단순 목록으로 보여주는 데서 끝나지 않고,
실제 초회 영업에 필요한 다음 정보를 한 화면에서 확인합니다.

- 대표전화
- 등록특허 보유 여부
- 순고용 증가
- 업체별 영업주제
- 추천등급과 추천사유
- 초회 영업전화 스크립트

기존 DB 연결·SQL 다운로드 화면은 삭제하지 않고 관리자 설정 안으로
정리해, 일반 사용자는 `수집 → 분석 → 초회전화` 순서만 따라가도록
UI를 단순화했습니다.

## 수정 파일과 영향 범위

- 수정: `prospect_db_center.py`
  - 후보표 연락처 자동표시, 특허·고용 필터, 영업주제와 스크립트
  - DB 기술설정 접기, 저장후보 작업버튼 단순화
- 수정: `prospect_db_repository.py`
  - 기존 영업후보의 `source_data`에 v9.7.1 영업분석 보존
- 수정: `localdata_contact_client.py`
  - 빠른 대표전화 확인용 조회 서비스 수 제한 옵션
- 신규: `kipris_patent_client.py`
  - 회사명 기준 등록특허 조회 및 일치도 확인
- 신규: `sales_intelligence.py`
  - 연락처·특허·고용 신호, 추천등급 및 초회 스크립트 생성
- 수정: `VERSION.txt`

기존 CRM, 고객관리, 상담일지, 정책자금, 고용지원금, 직원현황,
기업관리, 국민연금 기본·상세조회와 v9.7.0 연락처 정밀보강 기능은
삭제하거나 대체하지 않습니다.

## 적용 전 확인

현재 프로젝트 버전이 반드시 `v9.7.0`이어야 합니다.

다음 두 SQL을 이전 버전에서 이미 실행했다면 다시 실행할 필요가 없습니다.

- `supabase_v960_prospect_db.sql`
- `supabase_v970_contact_enrichment.sql`

v9.7.1은 기존 영업후보의 `source_data` JSON에 분석 결과를 추가하므로
새 Supabase SQL이 없습니다.

## KIPRIS 특허 API 준비

특허 필터를 사용하려면 KIPRIS Plus 특허·실용 공개·등록공보 REST API의
서비스 키가 필요합니다. 이 API는 공공데이터포털에서 `LINK` 유형으로
안내되며 실제 활용신청과 키 발급은 KIPRIS Plus에서 진행됩니다.

1. 공공데이터포털에서
   `지식재산처_KIPRISPlus_특허_실용 공개_등록공보_REST API`를 찾습니다.
2. 상세화면의 `바로가기`를 눌러 KIPRIS Plus로 이동합니다.
3. KIPRIS Plus 회원가입·로그인 후 특허 공개·등록공보 Open API를
   활용신청합니다.
4. 승인 후 발급된 서비스 키를 복사합니다.
5. Railway 앱 서비스의 `Variables`에 다음 이름으로 등록합니다.

```text
KIPRIS_API_KEY=발급받은_KIPRIS_Plus_서비스키
```

6. Railway의 변경사항을 배포합니다.

국민연금 조회에 쓰는 `DATA_GO_KR_SERVICE_KEY`와 KIPRIS Plus 키는
별도 서비스 키입니다. KIPRIS 키가 없어도 연락처·고용증가 분석은
작동하지만, 특허 필터는 비활성화되고 특허확인 상태에 `KEY_MISSING`이
표시됩니다.

## Railway 환경변수

현재 기능 전체에 필요한 변수 이름은 다음과 같습니다.

```text
DATA_GO_KR_SERVICE_KEY
KAKAO_REST_API_KEY
NAVER_CLIENT_ID
NAVER_CLIENT_SECRET
KIPRIS_API_KEY
```

키 값은 코드, ZIP 또는 GitHub에 입력하지 않습니다.

## 적용 방법

1. ZIP 내용을 기존 `정책자금자동화` 폴더에 바로 덮어 풉니다.
   ZIP 안에는 별도의 상위 패치 폴더가 없습니다.
2. `RUN_UPDATE_v9.7.1.bat`를 실행합니다.
3. 다음 결과를 확인합니다.

```text
UPDATE_OK
VERSION=v9.7.1
CANDIDATE_CONTACT=VISIBLE
PATENT_FILTER=ENABLED
EMPLOYMENT_SIGNAL=ENABLED
FIRST_CALL_SCRIPT=ENABLED
DB_SCHEMA=UNCHANGED
PY_COMPILE=OK
RUNTIME_SMOKE_TEST=OK
```

4. 동봉된 Git 명령어로 GitHub `main`에 업로드합니다.
5. Railway 배포 완료 후 `영업후보DB`를 엽니다.

## 새 화면 사용 순서

### 1. 외부 데이터 연결 확인

상단 `영업정보 데이터`에서 카카오, 네이버, 승인 인허가 API와
KIPRIS 특허의 등록상태를 확인합니다.

### 2. 사업장 수집 및 자동분석

처음에는 조회 건수를 10건으로 놓고 다음 버튼을 실행합니다.

```text
사업장 수집 및 영업정보 자동완성
```

최종 후보 중 최대 20개 업체에 대해 대표전화와 등록특허를 확인하고,
국민연금 신규취득자·상실가입자 차이로 순고용증가를 계산합니다.

후보표에서 다음 항목을 바로 확인할 수 있습니다.

- `대표전화`: 카카오 로컬 또는 승인 인허가 API에서 확인한 번호
- `특허등록`: 출원인명이 일치하고 등록번호가 확인된 특허 수
- `특허확인`: 특허 API 연결·조회 상태
- `순고용증가`: 신규취득자 수 - 상실가입자 수
- `영업주제`: 업체 신호에 맞는 상담 주제
- `추천등급`: 연락 가능성과 영업 신호를 합산한 A/B/C 등급
- `초회 전화 스크립트`: 확인된 사실을 근거로 만든 첫 통화 문구

### 3. 추천 필터

- `등록특허 보유업체만`: 특허 관련 정책자금·기술보증·연구개발
  및 세제검토 상담 주제가 있는 업체
- `순고용 증가업체만`: 최근 고용지원금 사전검토 주제가 있는 업체
- `추천등급`: 연락처와 복수 영업 신호가 있는 업체를 우선 확인

### 4. 저장된 영업후보 관리

- `연락처·특허·영업주제 분석`
  - 빠른 대표전화, 등록특허, 고용신호, 추천등급과 스크립트를 생성
- `이메일·홈페이지 정밀 보강`
  - 기존 v9.7.0 기능으로 네이버 검색과 공식 홈페이지까지 확인하고
    전화·이메일·홈페이지를 연락처DB에 저장

DB 연결점검과 SQL 다운로드는 `관리자 설정 · DB 연결 상태` 안에
그대로 보존되어 있습니다.

## 영업 스크립트 원칙

- 특허 보유는 혜택 확정이 아니라 상담 가능성을 찾는 신호입니다.
- 고용증가는 지원금 확정이 아니라 대상자 연령, 고용형태,
  채용경로와 고용유지기간을 확인하기 위한 신호입니다.
- 스크립트에는 확인된 공개정보와 확인 질문만 사용합니다.
- 실제 지원금, 세액공제, 정책자금 또는 기술보증 가능 여부는
  기업별 증빙과 최신 공고를 검토한 뒤 확정합니다.
- 수신거부 요청이 있으면 기존 연락처DB의 `do_not_contact` 정책을
  우선 적용합니다.

## 자동 백업과 롤백

- 적용 전 기존 파일은
  `_update_backups/v9.7.0_before_v9.7.1_날짜시간`에 백업됩니다.
- 새 파일도 별도로 추적합니다.
- `py_compile` 또는 실행검사 실패 시 기존 파일은 복구되고,
  이번 패치에서 생성한 새 파일은 제거됩니다.
- `.git`과 Git 연결은 수정하지 않습니다.

## GitHub 업로드

PowerShell에서 기존 프로젝트 폴더로 이동한 후
`GITHUB_UPLOAD_COMMANDS_v9.7.1.txt`의 명령어를 순서대로 실행합니다.

`git pull --rebase origin main`에서 충돌이 발생하면 강제 푸시하지 말고
충돌 파일을 확인한 뒤 해결해야 합니다.
